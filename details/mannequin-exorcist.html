01

생존 상태 FSM — 네트워크 동기화

퇴마사의 생존 상태는 5단계로 전환된다. 이 상태가 모든 클라이언트에 정확히 동기화돼야 올바른 UI·애니메이션·이동 제한이 표시된다. Photon Fusion 2의 [Networked] 프로퍼티로 State Authority(서버)가 단일 진실 공급원이 되게 했다.

  Normal ──피격──▶ Injured ──피격──▶ Downed ──방치 60초──▶ Dead
                              │                │
                           치료/구출          마네킹화
                              │                │
                           Normal          Mannequin ──방치 ~180초──▶ Dead
                                               │
                                            팀원 구출
                                               │
                                           Injured (WasRescued=true → 다음 피격 즉사)
PlayerStatusController.cs
// 상태 + 타이머 전부 [Networked] → State Authority만 수정, 모든 클라이언트 자동 반영
[Networked] public PlayerState State { get; set; }
[Networked] public float BleedOutRemaining { get; set; }
[Networked] private TickTimer HitBoost { get; set; }
[Networked] public bool WasRescued { get; set; }   // 구출 이력 — true면 다음 피격 즉사
[Networked] public bool IsBeingHelped { get; set; } // 치료/구출 중 → 중복 시작 차단

// 이동속도 배율: 상태에 따라 ExorcistController가 곱해서 사용
public float SpeedMultiplier => State switch {
    PlayerState.Injured  => IsHitBoosted ? 1.2f : 0.85f,  // 피격 직후 2초 부스트
    PlayerState.Downed   => 0.35f,
    PlayerState.Mannequin or PlayerState.Dead => 0f,
    _                    => 1.0f
};

출혈 타이머는 Downed=1배속, Mannequin=1/3배속으로 다르게 흐른다. 구출된 적이 있는 플레이어는 WasRescued=true가 세팅되고, 다음 피격 시 단계를 건너뛰고 즉사한다.

02

RPC 설계 — Race Condition과 중복 처리 방지

Heal이나 TakeDamage가 여러 클라이언트에서 동시에 호출되면 상태가 꼬인다. Photon Fusion 2에서 이를 막는 방법은 모든 상태 변경을 State Authority(서버)에서만 실행하고, 클라이언트는 RPC로 요청만 보내는 것이다.

PlayerStatusController.cs
// 상태 변경 함수: HasStateAuthority 가드로 서버에서만 실행
public void TakeDamage() {
    if (!Object.HasStateAuthority) return;
    // ...
}

// 팀원 치료: 동일 tick 중복 호출 방지 (LastHealTick 비교)
public void Heal() {
    if (!Object.HasStateAuthority) return;
    int tick = Runner.Tick;
    if (tick == LastHealTick) return; // 같은 틱 중복 호출 차단
    LastHealTick = tick;
    // ...
}

// RPC: 호출자(All/InputAuthority) → StateAuthority 단방향
[Rpc(RpcSources.InputAuthority, RpcTargets.StateAuthority)]
public void RPC_TakeDamage() => TakeDamage();

[Rpc(RpcSources.All, RpcTargets.StateAuthority)]
public void RPC_HealByTeammate() => Heal();

[Rpc(RpcSources.All, RpcTargets.StateAuthority)]
public void RPC_HitByMannequin() => TakeDamage(); // 마네킹 측에서 호출

같은 틱에 두 팀원이 동시에 RPC_HealByTeammate를 보내도 LastHealTick 비교로 한 번만 처리된다. FindObjectsOfType 대신 정적 HashSet으로 팀원 목록을 관리해 탐색 비용도 제거했다.

PlayerStatusController.cs
// FindObjectsOfType 대체 — Spawned/Despawned에서 관리
public static readonly HashSet<PlayerStatusController> All
    = new HashSet<PlayerStatusController>();

public override void Spawned()  => All.Add(this);
public override void Despawned(...) => All.Remove(this);
03

Vault — 네트워크 이동의 바닥 뚫림 버그

창틀·행거를 넘는 Vault 액션에서 가장 오래 막혔던 버그는 "넘는 도중 바닥을 뚫고 낙사"하는 문제였다. Photon Fusion 2의 NetworkCharacterController(NCC)가 내부적으로 중력을 계산하면서, 이동 중에 NCC 속도를 수정하면 충돌했다.

문제

Vault 이동 중 매 틱마다 _ncc.Teleport()와 transform.position을 동시에 수정하면, NCC 내부 중력 판정이 캐릭터를 바닥 아래로 밀어낸다.

해결

이동 중에는 transform.position만 수정하고, NCC는 건드리지 않는다. 도착 시점에만 _ncc.Teleport()로 한 번에 동기화.

VaultController.cs
private void TickVault() {
    VaultElapsed += Runner.DeltaTime;
    float t = Mathf.Clamp01(VaultElapsed / vaultDuration);
    float curveT = t * t * (3f - 2f * t);  // smoothstep — 부드러운 출발·정지

    Vector3 pos = Vector3.Lerp(VaultStart, VaultTarget, curveT);
    if (IsArcVault)
        pos.y += Mathf.Sin(t * Mathf.PI) * arcHeight; // 행거: 포물선 아크

    // 핵심: NCC 중력 판정 우회 — transform만 수정, NCC Teleport는 도착 후 1회만
    transform.position = pos;

    if (t >= 1f) CompleteVault();
}

private void CompleteVault() {
    transform.position = VaultTarget;
    _ncc.Teleport(VaultTarget);   // 도착 시점에만 네트워크 동기화
    _ncc.Velocity = Vector3.zero;
    _cc.enabled = true;
    IsVaulting = false;
}

추가로 Vault 경로 위에 있는 다른 플레이어를 수직 방향으로 밀어내는 로직도 구현했다. 경로(선분)와 플레이어 사이의 최단 거리로 판정하고, 어느 쪽에 있는지에 따라 방향을 결정한다.

04

치료·구출 — F키 홀드 + 스킬체크

치료(F홀드 8초)와 구출(F홀드)은 동일한 키를 쓰지만 서로 다른 컨트롤러가 처리한다. DefaultExecutionOrder로 실행 순서를 명시적으로 제어해서, 치료·구출이 동시에 시작되는 문제를 해결했다.

VaultController.cs / HealingController.cs
// VaultController: 먼저 실행해서 Vault 중이면 하위 컨트롤러가 F키 처리 안 하게
[DefaultExecutionOrder(-1)]
public class VaultController : NetworkBehaviour { ... }

[DefaultExecutionOrder(0)]
public class HealingController : NetworkBehaviour {
    // Vault 중이거나 이미 치료 중이면 진입 안 함
    if (_vault.IsVaulting || _rescue.IsRescuing) return;
}

치료 중간에 랜덤 위치에서 스킬체크(Q키 반응)가 발생한다. 성공하면 치료 게이지 보너스, 실패하면 패널티가 없는 대신 성공 시 보너스만 있는 방식으로 난이도를 조절했다.

HealingController.cs
// 치료 게이지 구간별 스킬체크 위치 (랜덤 2~3개)
private float[] _skillCheckThresholds;  // [0.3f, 0.6f, 0.85f] 같은 형태
private int _skillCheckIndex;
private bool _waitingForSkillCheck;

// FixedUpdateNetwork에서 매 틱 게이지 체크
if (_skillCheckIndex < _skillCheckThresholds.Length
    && Progress >= _skillCheckThresholds[_skillCheckIndex]) {
    _waitingForSkillCheck = true;  // 스킬체크 창 표시
    // Q키 입력 시 성공 처리 → 게이지 보너스
}

RESULT

결과 · 배운 점

  • 4인 멀티플레이 환경에서 퇴마사 3명 + 마네킹 1명이 동시에 상호작용하는 복잡한 상태를 안정적으로 동기화.
  • Photon Fusion 2의 State Authority / Input Authority 분리 모델이 Race Condition 방지에 어떻게 작동하는지 직접 구현하며 체득.
  • LastHealTick 패턴으로 같은 틱 중복 RPC를 차단 — 멀티플레이 특유의 동시성 문제를 네트워크 레벨에서 해결.
  • NCC와 transform.position의 충돌 — "왜 바닥을 뚫나"를 직접 추적해서 원인을 찾고 한 줄 수정으로 해결한 경험.
  • DefaultExecutionOrder로 컨트롤러 간 실행 순서를 명시 — 암묵적 의존성을 코드로 문서화하는 습관.
