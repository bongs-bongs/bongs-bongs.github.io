01

전체 구조 — BehaviorTree 계층

NPC 하나가 가게에 들어오는 순간부터 나가기까지, 18개의 BT Task · Service · Decorator가 순서대로 실행된다. 각 Task는 단일 책임 원칙을 지켜 테스트·재사용이 쉽게 설계했다.

  [ SpawnManager → NPC 생성 ]
          │
          ▼
  BT_Guest ─────────────────────────────────────────
  │
  ├─ RollShopEntry     매장 진입 여부 확률 판정
  ├─ BeginNPCSession   Blackboard 초기값 세팅
  ├─ FindWanderPoint   빈 기계 탐색 (MachineCache 활용)
  ├─ RotateToTarget    기계 앞에서 방향 조정
  ├─ DecideStayOrLeave MachineStayChance / ShopStayChance 계산
  ├─ PlayAnim          뽑기 시퀀스 애니메이션
  ├─ LogVisitCount     SuccessCount, FailStreak 갱신
  ├─ [bShouldExit?] ──▶ FindExit → DestroySelf
  │
  └─ [흑화 조건 충족] ──▶ SwapBehaviorTree(BT_Jinsang_*)
                          같은 BlackboardAsset 재사용 → 상태 유지
02

동적 생성 — 기계 수에 반응하는 스폰

플레이어가 기계를 추가하면 NPC가 더 자주, 더 많이 온다. 하드코드된 값이 아니라 현재 기계 수를 실시간으로 읽어 SpawnInterval과 MaxConcurrent를 계산하게 했다.

문제

기계가 1대일 때와 10대일 때 같은 속도로 NPC가 스폰되면, 기계 추가가 게임플레이에 의미 없음.

해결

기계 1대당 SpawnInterval 차감, MaxConcurrent 증가. 기계 추가/제거 시 타이머 즉시 재시작.

CMSGuestSpawnManager.h
// 기계 1대 추가 → 스폰이 빨라지고 동시 허용 NPC 수 증가
UPROPERTY(EditAnywhere, Category = "Spawn|Config")
float SpawnIntervalReductionPerMachine = 0.1f;

UPROPERTY(EditAnywhere, Category = "Spawn|Config")
float MinSpawnInterval = 2.0f;          // 하한선 — 무한 빨라짐 방지

UPROPERTY(EditAnywhere, Category = "Spawn|Config")
int32 MaxConcurrentBonusPerMachine = 1;  // 기계 1대당 +1명

// 실제 계산: SpawnInterval - MachineCount * Reduction, MinSpawnInterval로 clamp
float GetEffectiveSpawnInterval() const;

카테고리(손님/통행인/진상)별로도 동시 허용 수를 별도 추적한다. enum 항목을 추가하면 컴파일 타임에 즉시 에러가 나서 카운터 배열 동기화를 강제했다.

CMSGuestSpawnManager.h
static constexpr int32 NumNpcCategories = static_cast<int32>(ECMSNpcCategory::Count);
int32 ActiveCategoryCounts[NumNpcCategories] = {};

// enum에 항목 추가 시 switch 동기화를 컴파일 에러로 강제
static_assert(NumNpcCategories == 3,
    "ECMSNpcCategory 항목 추가 시 CategoryToIndex switch 도 갱신하고 본 assert 값 동기화하세요.");
03

BT Swap — 같은 상태로 진상 전환

일반 손님이 조건을 만족하면 진상(Villain)으로 전환된다. 새 NPC를 스폰하는 대신 같은 BlackboardAsset을 유지한 채로 BehaviorTree만 교체해서, FailStreak·CurrentMachine 같은 상태가 그대로 이어진다.

문제

진상 전환 시 기존 NPC를 지우고 새로 스폰하면 상태가 초기화되고, 전환 순간 빈틈이 생겨 어색함.

해결

BlackboardAsset은 그대로 두고 SwapBehaviorTree(BT_Jinsang_*)만 호출. BB 값이 유지된 채로 새 행동 트리가 실행됨.

CMSGuestAIController.h
// 같은 BlackboardAsset 재사용 → BB 상태(FailStreak, CurrentMachine 등) 유지.
UFUNCTION(BlueprintCallable, Category = "AI")
bool SwapBehaviorTree(UBehaviorTree* NewBT);

// 흑화(진상 전환) 시 BB 키 set + SwapBehaviorTree 호출
// inline const FName bIsVillain("bIsVillain"); ← BB 키 SSOT

Blackboard 키는 별도 네임스페이스(CMSGuestBB)에 단일 출처(SSOT)로 정의해서, 오타로 인한 런타임 에러를 차단했다.

CMSGuestBlackboardKeys.h
namespace CMSGuestBB {
    inline const FName FailStreak(TEXT("FailStreak"));     // 연속 실패 횟수
    inline const FName bIsVillain(TEXT("bIsVillain"));     // 흑화 여부
    inline const FName bWaitingForBlocker(TEXT("bWaitingForBlocker")); // 길막 대기
    inline const FName CurrentMachine(TEXT("CurrentMachine")); // 점유 중인 기계
    // ... 총 20개 키
}
04

MachineCache — BT 매 프레임 탐색 비용 제거

NPC가 빈 기계를 찾을 때마다 GetAllActorsOfClass를 호출하면, NPC가 10명이 동시에 기계를 탐색할 때 매 프레임마다 월드 전체를 순회한다. 정적 캐시 + 월드 교체 시 자동 무효화로 이 비용을 제거했다.

CMSGuestMachineCache.h
// 매 BT 실행마다 GetAllActorsOfClass 호출 회피용 머신 캐시.
// 멀티-PIE 시 월드 교대마다 invalidate. 단일 플레이/Shipping 무관.
class FCMSGuestMachineCache {
public:
    static const TArray<TWeakObjectPtr<ACMSClawMachineBase>>&
        GetMachines(UWorld* World);  // World 다르면 자동 재빌드

    static void InvalidateCache();   // 기계 추가/제거 시 호출
private:
    static TArray<TWeakObjectPtr<ACMSClawMachineBase>> CachedMachines;
    static TWeakObjectPtr<UWorld> CachedWorld;
};

TWeakObjectPtr를 써서 기계가 월드에서 제거되면 자동으로 포인터가 무효화되고, 다음 탐색 시 재빌드된다. 호출자가 유효성을 확인하는 책임 제도를 명확히 해서 순환참조도 차단했다.

05

Settlement 진입 → 전체 NPC 자연 퇴장

영업 시간이 끝나면 가게 안의 모든 NPC가 자연스럽게 나가야 한다. 새 BT로 교체하거나 강제 Destroy하지 않고, BB 키 하나(bShouldExit=true)를 set해서 각 NPC의 BT가 다음 사이클에 Exit 분기로 자연 진입하게 했다.

CMSGuestSpawnManager.h
// Settlement 진입 시 BT Exit 분기로 보내 자연 퇴장 트리거.
void ForceAllGuestsLeave();

// BB: bShouldExit = true → BT 다음 사이클에 FindExit → DestroySelf
// inline const FName bShouldExit("bShouldExit"); ← BB 키

이 방식 덕분에 NPC가 멈추거나 텔레포트하는 현상 없이, 현재 행동(뽑기 시퀀스)을 마친 뒤 출구로 이동해서 사라진다.

결과 · 배운 점

  • 게임 내 동시에 10+ NPC가 각자 독립적으로 BT를 실행하면서 자연스러운 가게 분위기 연출.
  • 코드 변경 없이 데이터(GuestTypeConfig)만 바꿔 손님 성격 조절 → 레벨 디자이너와의 협업 마찰 최소화.
  • static_assert로 enum 추가 → 컴파일 타임 에러 → 런타임에 조용히 터지는 버그를 사전 차단.
  • BT Swap 패턴으로 진상 전환 시 상태 연속성 보장 — "전환 순간 어색함" 완전 제거.
  • AI 시스템에서도 데이터 → 코드 분리가 유지보수에 얼마나 중요한지 실감.
