WeGo LIMO Pro 플랫폼 위에서 차선 인식 → 장애물 감지 → 통합 주행 제어까지 전 파이프라인을 ROS 노드로 설계했다. 각 인식 결과가 토픽으로 분리되고 중앙 컨트롤러가 우선순위에 따라 통합 판단하는 구조가 핵심이다.

My Part · 인식·제어 파이프라인 ROS1 (Noetic) Python OpenCV · HLS LiDAR (LaserScan) YOLO

전체 파이프라인을 차선 인식 노드 / LiDAR 긴급정지 노드 / 통합 제어 노드 3개로 분리 설계하고 구현했다. 각 노드는 ROS 토픽으로만 통신하며 독립적으로 실행·재시작 가능하다.

01

전체 구조 — ROS 토픽 파이프라인

센서 데이터에서 주행 명령까지 단방향 토픽 파이프라인으로 설계했다. 각 노드가 독립적으로 동작하고, 제어 노드는 모든 인식 결과를 구독해서 우선순위에 따라 최종 cmd_vel을 결정한다.

  [카메라] ──▶ lane_detect   ──▶ /limo/lane_x        (차선 무게중심 x)
                crosswalk_detect ▶ /limo/crosswalk_y   (횡단보도까지 거리 y)
  [LiDAR]  ──▶ e_stop        ──▶ /limo/lidar_warning  (Safe / Warning)
                               ──▶ /limo/obstacle_angle (장애물 방향)
  [YOLO]   ──────────────────▶ /limo/yolo_object     (신호등·표지판 클래스)
                                         │
                                         ▼
                                    control.py
                                    /cmd_vel (Twist) → LIMO 주행

토픽 기반 분리 덕분에 차선 인식 파라미터만 바꾸거나 LiDAR 노드만 재시작할 때 다른 노드에 영향이 없다. Dynamic Reconfigure로 런타임 튜닝도 지원한다.

02

차선 인식 — HLS 색 공간 + 모멘트

YOLO가 HLS 색 공간을 기준으로 동작하기 때문에, 차선 인식도 HLS로 통일해 노란·흰색 차선을 동시에 검출했다. 검출된 영역의 무게중심(Moment) x좌표를 기준점과 비교해 조향 오차를 계산했다.

lane_detect.py
def colorDetect(self, _img):
    hls = cv2.cvtColor(_img, cv2.COLOR_BGR2HLS)  # 조명 변화에 강한 HLS 변환
    mask_yellow = cv2.inRange(hls, self.YELLOW_LANE_LOW_TH, self.YELLOW_LANE_HIGH_TH)
    mask_white  = cv2.inRange(hls, self.WHITE_LANE_LOW_TH,  self.WHITE_LANE_HIGH_TH)
    return cv2.bitwise_or(mask_yellow, mask_white)  # 두 마스크 합산

def calcLaneDistance(self, _img):
    try:
        M = cv2.moments(_img)
        self.x = int(M['m10'] / M['m00'])  # 무게중심 x = 차선 중앙
    except:
        self.x = -1   # 차선 미검출 → control에서 직진 유지
    return self.x

ROI는 이미지 하단 30px만 잘라 연산량을 줄이고, 파라미터(임계값·기준점)는 Dynamic Reconfigure로 주행 중에 바로 조절할 수 있게 했다.

03

LiDAR 긴급정지 — 방향까지 판단

단순히 "장애물 있음/없음"만 판정하면 회피 방향을 모른다. 좌측(음수 각도)과 우측(양수 각도)을 분리 카운팅해서 어느 쪽에 장애물이 더 많은지 방향까지 퍼블리시했고, 제어 노드가 이 방향을 보고 반대쪽으로 회피한다.

e_stop.py
for i, angle in enumerate(angle_deg):
    distance = _data.ranges[i]
    if 0.0 < distance <= 0.5:       # 50cm 이내 점만 카운트
        if angle > 0:   cnt_positive += 1   # 우측
        elif angle < 0: cnt_negative += 1   # 좌측

# 더 많은 쪽이 장애물 방향 (음수 = 좌측)
obstacle_angle = cnt_positive if cnt_positive >= cnt_negative else -cnt_negative

if cnt_positive + cnt_negative >= self.E_STOP_COUNT:  # 노이즈 필터
    self.warn_pub.publish("Warning")
    self.angle_pub.publish(obstacle_angle)

제어 노드는 이 방향 정보를 받아 장애물이 우측이면 좌회전, 좌측이면 우회전으로 회피한다. 단순 정지 대신 능동적 회피가 가능해진다.

control.py — drive_callback
if self.e_stop == "Warning":
    if self.obstacle_angle > 0:   # 장애물이 우측 → 좌로 회피
        drive_data.angular.z = 1.0
    else:                          # 장애물이 좌측 → 우로 회피
        drive_data.angular.z = -1.0
    drive_data.linear.x = 0.2   # 저속으로 계속 이동하며 회피
04

통합 제어 — 우선순위 기반 분기

여러 인식 결과가 동시에 들어올 때 어떤 걸 우선할지가 핵심이다. 위험도 순서로 우선순위를 명시적으로 정의해서, 신호등과 장애물이 동시에 감지돼도 안전하게 처리된다.

  우선순위 (높음 → 낮음)
  ① LiDAR Warning     → 저속 회피 주행
  ② 신호등 Red/Yellow → 완전 정지
  ③ 서행 표지판       → 반속 주행
  ④ 보행자 표지판     → 5초 정지 (쿨다운 20초)
  ⑤ 차선 이탈 보정    → 비례 조향 (distance_to_ref × LATERAL_GAIN)
  ⑥ 정상              → BASE_SPEED 직진
control.py — drive_callback
# 기본 조향: 차선 무게중심 오차 × Gain (비례 제어)
drive_data.angular.z = self.distance_to_ref * self.LATERAL_GAIN

# 우선순위 순서로 override
if self.e_stop == "Warning":          # ① 가장 우선
    ...
elif self.yolo_object in ("yellow", "red"):  # ②
    drive_data.linear.x = 0.0
elif self.yolo_object == "slow":         # ③
    drive_data.linear.x = self.BASE_SPEED / 2
elif self.yolo_object == "pedestrian" \
    and self.is_pedestrian_stop_available \
    and self.bbox_size[0] > self.PEDE_STOP_WIDTH:  # ④ bbox 크기로 근접 판정
    drive_data.linear.x = 0.0
    self.is_pedestrian_stop_available = False     # 20초 쿨다운
else:                                             # ⑤⑥ 정상 주행
    drive_data.linear.x = self.BASE_SPEED
05

Ackermann / Differential 모드 전환

LIMO는 Ackermann(조향) 모드와 Differential Drive 모드 두 가지를 지원한다. 같은 angular.z 값이라도 모드에 따라 실제 회전량이 다르기 때문에, 모드를 감지해서 출력 변환을 다르게 했다.

control.py
if self.limo_mode == "diff":
    self.drive_pub.publish(drive_data)   # Differential: angular.z 그대로
elif self.limo_mode == "ackermann":
    if drive_data.linear.x == 0:
        drive_data.angular.z = 0        # 정지 시 조향각 리셋 (타이어 중립)
    else:
        # Ackermann 조향각 변환: tan(z/2) × v / wheelbase
        drive_data.angular.z = \
            math.tan(drive_data.angular.z / 2) * drive_data.linear.x / self.LIMO_WHEELBASE
    self.drive_pub.publish(drive_data)

Ackermann 모드에서 정지할 때 angular.z를 0으로 리셋하지 않으면 타이어가 꺾인 채로 정지해 재출발 시 급격하게 꺾이는 문제가 있었다. 이를 명시적으로 처리했다.

06

YOLO 인식 결과 시간 보정

YOLO 인식이 잠깐 끊기면 마지막 클래스가 계속 유지돼 불필요하게 정지 상태가 지속된다. 마지막 인식 시각을 기록하고 3초가 지나면 "green"으로 초기화해서, 인식 끊김으로 인한 영구 정지를 방지했다.

control.py — drive_callback
# YOLO 인식이 3초 이상 끊기면 green으로 초기화 (인식 노이즈 방지)
if self.yolo_object != "green" \
        and self.calcTimeFromDetection(self.yolo_object_last_time) > 3.0:
    self.yolo_object = "green"
    self.bbox_size = [0, 0]

보행자 표지판도 마찬가지로 bbox 크기(픽셀 너비)로 근접 여부를 판단해서, 멀리서 보행자 표지판을 인식했을 때 불필요하게 정지하지 않게 했다.

RESULT

결과 · 배운 점

  • 차선 추종·신호등·횡단보도·LiDAR 장애물 회피를 통합한 자율주행 경진대회 대상 수상.
  • ROS 토픽으로 노드를 분리하면 디버깅 시 특정 노드만 재시작할 수 있어 개발 속도가 빨라짐을 직접 체감.
  • YOLO와 색 공간을 HLS로 통일 — 시스템 전체에서 색 공간을 맞추는 것이 파이프라인 일관성에 중요함을 체감.
  • 단순 정지가 아닌 방향까지 판단한 회피 — "장애물이 어디에 있는가"를 데이터에서 뽑아내는 것이 제어 품질을 바꿈.
  • Ackermann 모드에서 정지 시 조향각 리셋 버그 — 물리 시스템 특성을 이해하고 코드에 반영하는 것이 중요함.
