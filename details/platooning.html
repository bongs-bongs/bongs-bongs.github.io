군집주행 Platooning
— ESP32 통신 게이트웨이

RC카 3대가 앞차를 따라가는 V2V 군집주행 시스템에서, 차량 간(MQTT) ↔ 차량 내(CAN) 통신을 잇는 ESP32 게이트웨이를 설계·구현했다. 실시간 제어에서 "반 박자 늦고 튀는" 문제를 통신 계층에서 어떻게 잡았는지가 핵심.

My Part · ESP32 통신 ESP32 (Arduino) MCP2515 CAN MQTT (PubSubClient) RPi4 · STM32 연동

전체 시스템은 RPi4(주행 제어·비전) + STM32(센서·모터) + ESP32(통신) 3보드로 구성된다. 이 문서는 내가 맡은 ESP32 통신 게이트웨이를 다룬다 — 다른 차량의 상태를 받아 내 차의 RPi4에게 전달하고, 내 차의 센서/제어값을 다시 군집에 방송하는 중개기다.

01

왜 통신을 ESP32에 분리했나

처음엔 RPi4 하나가 MQTT(WiFi)와 CAN을 모두 처리하게 하려 했다. 하지만 RPi4는 비전 추론(TFLite)과 주행 제어를 동시에 돌려야 해서, 여기에 WiFi 송수신 + 실시간 CAN 폴링까지 얹으면 제어 주기가 흔들렸다.

문제

RPi4가 비전 + 제어 + 통신을 다 맡으니 실시간 CAN 수신이 밀리고, MQTT 지연이 제어 루프에 그대로 전파됨.

해결

통신을 ESP32 게이트웨이로 분리. ESP32가 MQTT↔CAN 브리지만 전담하고, RPi4는 CAN으로 정제된 "앞차 정보"만 받아 제어에 집중.

  [ 다른 차량들 ]
        │  MQTT (WiFi, 브로커=로컬 PC)
        ▼
  ┌─────────────┐   CAN(0x100/0x110)   ┌──────────┐
  │   ESP32     │ ───────────────────▶ │   RPi4   │  주행 제어 + 비전
  │  (게이트웨이) │ ◀─────────────────── │          │
  └─────────────┘   CAN(0x140 조향)     └──────────┘
        ▲  CAN(0x120 속도 / 0x130 초음파)
        │
  ┌──────────┐
  │  STM32   │  엔코더·초음파·모터
  └──────────┘
02

차량 체인 구조 — 앞차만 구독

리더(car1) → 팔로워(car2) → 팔로워(car3) 순서로, 각 차량은 자기 바로 앞차의 토픽만 구독한다. 같은 펌웨어를 쓰고 MY_CAR_ID/FRONT_CAR_ID 두 줄만 바꿔 각 차에 플래시했다.

esp32_bridge.ino
/* car1: MY=1 FRONT=0 (앞차 없음, 구독 안 함)
   car2: MY=2 FRONT=1
   car3: MY=3 FRONT=2                         */
#define MY_CAR_ID     3
#define FRONT_CAR_ID  2

// 내 차 상태는 publish, 앞차 토픽은 subscribe
topic_my_ctrl    = "car" + String(MY_CAR_ID)    + "/ctrl_50ms";
topic_front_ctrl = "car" + String(FRONT_CAR_ID) + "/ctrl_50ms";
if (FRONT_CAR_ID != 0) mqtt.subscribe(topic_front_ctrl.c_str());
03

"반 박자 늦고 튀는" 문제 — Last-value + Stale timeout

MQTT는 브로커를 거치고 TCP 재전송까지 있어, 단순히 받은 순서대로 큐에 쌓아 처리하면 오래된 패킷이 뒤늦게 도착해 제어가 튀었다. 그래서 큐잉을 버리고, 수신 콜백은 "최신값만 덮어쓰고 수신 시각만 기록"하게 했다.

esp32_bridge.ino — onMqtt()
// 앞차 정보 수신 → 저장만(덮어쓰기). 큐잉·블로킹 없음
void onMqtt(char* topic, byte* payload, unsigned int length) {
  // ... JSON 파싱 ...
  if (data[0] != (byte)FRONT_CAR_ID) return;   // 앞차 ID만 통과
  if (is_front_ctrl) {
    for (int i = 0; i < 6; i++) front_ctrl_buf[i] = data[i];
    front_ctrl_rx_ms = now;       // 수신 시각 기록 (stale 판정용)
    front_ctrl_valid = true;
  }
}

그리고 CAN으로 내보낼 때, 너무 오래된 데이터(250ms↑)는 버리고 50ms 주기로만 포워딩한다. 이렇게 "항상 최신 한 개"만 흘려보내니 지연·튐이 사라졌다.

esp32_bridge.ino — forward_front_ctrl_if_ready()
if (!front_ctrl_valid) return;
if ((now - front_ctrl_rx_ms) > FRONT_CTRL_STALE_MS)   return; // 250ms↑ → 버림
if (now - last_front_ctrl_fwd_ms < FRONT_CTRL_FWD_PERIOD_MS) return; // 50ms 주기

CAN.sendMsgBuf(CAN_ID_TO_RPI_FRONT_CTRL, 0, 6, data);  // → RPi4로 전달
04

네트워크·로그 폭주 막기 — 조건부 전송

초음파 값을 매 주기 그대로 쏘면 CAN 버스와 로그가 의미 없는 데이터로 가득 찼다. 그래서 (1) 위험 거리거나, (2) 유의미하게 변했거나, (3) 1초간 조용했으면(하트비트) 일 때만 전송하도록 했다.

esp32_bridge.ino — forward_front_ultra_if_ready()
// 위험거리면 무조건, 아니면 유의미 변화 OR 1초 하트비트일 때만
bool danger = (L <= FRONT_ULTRA_DANGER_TH) || (R <= FRONT_ULTRA_DANGER_TH);
bool significant =
    (uabs16(L, last_front_ultra_L) >= FRONT_ULTRA_SEND_DIFF_TH) ||
    (uabs16(R, last_front_ultra_R) >= FRONT_ULTRA_SEND_DIFF_TH);
bool force_hb = (now - last_front_ultra_send_ms) >= FRONT_ULTRA_FORCE_MS;

if (!(danger || significant || force_hb)) return;   // 셋 다 아니면 생략
CAN.sendMsgBuf(CAN_ID_TO_RPI_FRONT_ULTRA, 0, dlc, data);

덕분에 평상시 트래픽은 크게 줄면서도, 위험 상황에서는 즉시·확실히 전달되는 안전 마진을 확보했다.

05

ESP32에 맞춘 경량 JSON 파서

MQTT 페이로드는 JSON인데, ArduinoJson은 ESP32에서 동적 할당 부담이 있었다. 어차피 필요한 필드는 dlc와 data[] 둘뿐이라, 그 두 개만 긁어오는 미니 파서를 직접 짰다.

esp32_bridge.ino — json_get_array8()
// "data":[..] 배열만 직접 파싱 (동적 할당 없음)
static bool json_get_array8(const String& s, const char* key,
                            byte out[8], int& out_n) {
  String k = String("\"") + key + "\":[";
  int p = s.indexOf(k);
  if (p < 0) return false;
  // p부터 ']' 전까지 숫자만 추출해 out[]에 채움
  return (out_n > 0);
}
06

CAN ID 맵

ESP32를 가운데 두고 STM32·RPi4가 주고받는 CAN 프레임을 ID로 분리했다. 모든 프레임의 첫 바이트는 차량 ID라 버스에서 내 차 것만 필터링한다.

CAN ID방향내용
0x120STM32 → ESP32속도 (엔코더 기반)
0x130STM32 → ESP32초음파 L / R / 정면
0x140RPi4 → ESP32조향값
0x100ESP32 → RPi4앞차 속도 + 조향
0x110ESP32 → RPi4앞차 초음파
RESULT

결과 · 배운 점

  • RC카 3대가 앞차를 일정 간격으로 따라가는 군집주행 데모 동작.
  • 실시간성 요구 통신에서 "신뢰성(큐·재전송) ≠ 항상 좋음" — 늦게 오느니 버리는 last-value 설계가 제어 안정성에 더 유리하다는 걸 체감.
  • 임베디드 통신은 대역폭·로그를 의식한 조건부 전송이 안정성과 디버깅 모두에 중요.
  • 보드 역할 분리(통신/제어/센서)로 각 루프의 실시간성을 독립적으로 지킬 수 있었다.
